Feature 5: Incentives System

TASK: Implement Incentives System for Schema-Sync ERP
OBJECTIVES:
Implement a flexible incentives system that calculates and accrues daily incentives based on 
performance metrics, KPIs, and sales targets. Incentives are added to employee salaries during 
payroll processing with NO impact on accounting treasury (accrual-based, not paid separately).

KEY BUSINESS RULES:
1. Incentives are accrued DAILY based on performance metrics
2. Accrued incentives are included in next month's payroll as a salary component
3. NO separate journal entries for incentives (included in salary expense)
4. Incentives are NOT treasury-impacting (part of salary calculation)
5. Reversal of incentives when performance metrics change retroactively
6. Multi-currency support with conversion at accrual date

DATABASE SCHEMA (Drizzle ORM):
Tables to create:
1. incentive_schemes (id, company_id, name_en, name_ar, description, status,

2. incentive_rules (id, incentive_scheme_id, metric_type, target_value,

   - metric_types: Sales Amount, Units Sold, Invoices Created, Customers Acquired, etc.
   - incentive_types: Fixed Amount, Percentage of Target, Per Unit
   - calculation_methods: Achievement, Slab-based, Tiered
3. incentive_slab (id, incentive_rule_id, slab_number, from_percentage, to_percentage, 
   incentive_value, created_at)
   - Used for slab-based calculations (e.g., 0-50% → 100 EGP, 51-100% → 250 EGP, 100%+ → 500 EGP)
4. employee_incentive_assignments (id, employee_id, incentive_scheme_id, assigned_date, 
   end_date, status, created_at)
5. daily_incentive_accrual (id, employee_id, incentive_rule_id, accrual_date, 
   metric_value, target_value, achievement_percentage, accrued_amount, currency, 
   status, created_at)
6. monthly_incentive_summary (id, employee_id, month, total_accrued, included_in_payroll_record_id, 
   status, created_at)
7. incentive_accrual_reversal (id, daily_accrual_id, reversal_reason, reversal_date, 
   reversal_amount, created_by, created_at)
8. incentive_metrics (id, incentive_rule_id, metric_date, employee_id, metric_value, 
   source_document_id, source_type, created_at)
   - source_type: Sales Invoice, Purchase Order, Customer Record, Manual Entry, API

RELATIONSHIPS:
- incentive_schemes.company_id → companies.id
- incentive_rules.incentive_scheme_id → incentive_schemes.id
- incentive_slab.incentive_rule_id → incentive_rules.id
- employee_incentive_assignments.employee_id → employees.id
- employee_incentive_assignments.incentive_scheme_id → incentive_schemes.id

- incentive_metrics.incentive_rule_id → incentive_rules.id
- incentive_metrics.employee_id → employees.id

ACCRUAL CALCULATION LOGIC:
1. Daily Process:
   - Fetch incentive rules assigned to each employee
   - For each rule, calculate metric value for the day
   - Calculate achievement percentage = (metric_value / target_value) * 100
   - Look up slab or calculate incentive based on calculation_method
   - Create daily_incentive_accrual record with accrued_amount
   - Status: "Accrued"

2. Slab-Based Calculation Example:
   - Rule: Sales Target = 10,000 EGP/day
   - Slabs: 0-50% → 0 EGP, 51-75% → 100 EGP, 76-100% → 250 EGP, 100%+ → 500 EGP
   - If sales = 8,000 EGP (80% achievement) → 250 EGP incentive

3. Tiered Calculation Example:
   - Rule: Invoice creation incentive
   - 1-5 invoices → 50 EGP each = 250 EGP
   - 6-10 invoices → 75 EGP each = 450 EGP
   - 11+ invoices → 100 EGP each
   - If 8 invoices created → 250 + (3 × 75) = 475 EGP

4. Monthly Aggregation:
   - Sum all daily_incentive_accrual records for employee for the month
   - Create monthly_incentive_summary record
   - Status: "Pending Payroll"

5. Payroll Integration:
   - Query monthly_incentive_summary for payroll period
   - Add incentive_amount as line item in payroll_records
   - Update monthly_incentive_summary status to "Included in Payroll {payroll_record_id}"

6. Reversal:
   - If metric is corrected retroactively, create incentive_accrual_reversal
   - Recalculate monthly summary
   - If already in payroll, create negative adjustment in payroll record

BUSINESS LOGIC:

1. REQUEST VALIDATION:
   - Employee must have minimum salary (configurable, e.g., > 5,000 EGP)
   - Requested amount <= (monthly_salary * max_advance_percentage / 100)
   - Number of outstanding advances <= max_consecutive_advances
   - Reason field is required
   - Cannot request if already in deduction for previous advance

2. APPROVAL WORKFLOW:
   - Status: Pending → Approved → Active (if approved) OR Rejected
   - Manager reviews and approves/rejects
   - If approved: Create salary_advance_deductions schedule
   - If rejected: Store rejection_reason, notify employee
   - Approval notification sent to employee

3. DEDUCTION CALCULATION:
   - When payroll period starts after approval_date
   - Monthly deduction = approved_amount / repayment_tenure_months
   - Deduct from net salary or as separate line item (configurable)
   - Create salary_advance_deductions record for each month
   - Status changes: Active → Completed when fully repaid

4. OUTSTANDING BALANCE:
   - Balance = total_approved_advances - total_deductions - manual_payments
   - Multiple advances: Sum of all outstanding balances

5. MANUAL PAYMENT:
   - Employee can pay manually at any time
   - Creates salary_advance_deductions record with negative amount
   - Updates balance
   - No GL impact (cash receipt handled separately)

6. CANCELLATION:
   - Only admin can cancel approved advance
   - Cancels remaining deductions
   - Creates journal entry reversal

ACCOUNTING INTEGRATION:
On Advance Approval:
- Debit: 1200-Salary Advances (Asset account)
- Credit: 2102-Salary Advances Payable (Liability)

On Deduction from Payroll:
- Debit: 2102-Salary Advances Payable
- Credit: 1200-Salary Advances (Asset account)
- Net effect: Liability reduction

VALIDATION:
- Required: employee_id, requested_amount, advance_type, reason
- Requested_amount > 0
- Requested_amount <= max_allowed (50% of salary)
- Employee has minimum salary
- No more than max_consecutive_advances outstanding
- Status transitions valid (Pending → Approved/Rejected, etc.)
- Deduction amount matches calculation
- Manual payment <= outstanding balance

VALIDATION & ERROR HANDLING:
- 400: Invalid advance amount
- 409: Employee has reached max concurrent advances
- 422: Employee salary < minimum
- 404: Advance not found
- 403: Insufficient permissions

ERROR SCENARIOS:
- Employee requests advance, salary is low → 422
- Request amount > 50% monthly salary → 400
- Already 3 outstanding advances → 409
- Manager tries to approve advance outside their org → 403
